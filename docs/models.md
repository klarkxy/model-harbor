# 模型管理

ModelHarbor 把「上游真实模型」和「下游看到的模型名」解耦。管理员可以灵活定义暴露面，而不用让下游直接感知供应商细节。

---

## 核心概念

- **真实模型名（real model name）**：上游供应商 API 里的模型 ID，例如 `deepseek-chat`、`claude-3-5-sonnet-20241022`。
- **公共模型（Public Model）**：你希望下游客户端请求的模型名，例如 `ds-chat`、`claude-sonnet`。
- **候选（candidate）**：一个公共模型可以映射到多个 `(upstream key, real model name)` 候选，形成路由池。
- **模型组（Model Group）**：管理员自定义的集合，例如 `coder`、`fast`、`cheap`。下游可以像请求单个模型一样请求一个组。

---

## 命名规则

- 公共模型名和模型组名共享同一个全局命名空间，**不能重复**。
- 名称建议使用小写、数字、点号、下划线、中划线：`a-z0-9._-`。
- 客户端请求时模型名**不区分大小写**（例如 `GPT-4o`、`gpt-4o`、`Gpt-4o` 都会解析到同一模型），但后台统一按小写存储和匹配。
- 展示名（display name）可以使用中文或更友好的名字。

---

## 创建公共模型

1. 进入 **Public Models → New public model**。
2. 填写：
   - **Name**：下游请求的模型名，例如 `ds-chat`。
   - **Display name**：展示名，例如 `DeepSeek Chat`。
   - **Description**：可选，用于管理后台备注。
3. 添加候选：
   - 选择一条已创建的 upstream key。
   - 填写该 upstream key 下的真实模型名。
   - 设置 **Priority**：数字越小越优先（默认路由下先尝试）。
   - 设置 **Weight**：用于加权随机策略。
   - 启用/禁用该候选。
4. 保存。

一个公共模型至少要有一个启用的候选，否则下游请求无法路由。已存在但没有候选的模型通常返回配置校验错误；如果目标本身不存在则返回 404。

---

## 编排公共模型候选

公共模型创建后，可以在 **Public Models** 列表中进入候选编排面板。编排面板面向日常管理：

- 抓住表格最左侧的拖动符号调整候选顺序。
- 新增、删除、启用或禁用某个候选。
- 修改候选的 upstream key 和真实模型名。
- 查看候选对应的协议 / 供应商，以及上游是否禁用、冻结或健康降级。
- 点击 **Reset default order** 可清除该公共模型的手动编排，并重新按当前上游密钥顺序排列候选。

保存时，面板会把当前可见顺序自动写成 `10, 20, 30...` 这样的 priority，并为候选保留默认 weight。管理员通常只需要关心列表顺序；priority / weight 只是路由层保存顺序与未来负载策略的内部字段，当前不在编排表格中展示。

**上游密钥**列表本身也可以通过最左侧抓手排序。未手动编排过的公共模型会自动跟随上游密钥顺序作为默认候选顺序；已经手动编排过的公共模型不会被上游排序覆盖，后续新增候选只会追加到末尾。真实请求时还会经过配额、能力、协议、熔断、端点健康和 sticky 绑定检查；因此 sticky 命中或端点健康排序可能会把某个仍可用的候选提前。

---

## 候选优先级与路由策略

对于单个公共模型，默认使用 **线性优先级回退**：按 priority 从小到大依次尝试，第一个可用候选处理请求，失败则尝试下一个。

对于模型组，路由策略已统一为按成员顺序回退：列表中越靠上的成员优先级越高，请求时依次尝试，失败则跳到下一个。你可以在创建模型组时通过拖拽成员调整顺序。

> 会话粘性（sticky binding）会覆盖负载均衡策略。如果同一会话之前绑定到某个候选且仍可用，系统会优先复用。

---

## 创建模型组

1. 进入 **Model Groups → New model group**。
2. 填写名称、展示名、描述。
3. 添加成员：选择已创建的公共模型。
4. 通过拖拽成员调整顺序：越靠上优先级越高。
5. 保存。

> 创建上游密钥时不会自动创建模型组。模型组必须由管理员手动创建和维护。

---

## 自动发现模型

在创建上游密钥时，可以点击 **Fetch models** 从供应商拉取模型列表。拉取到的真实模型名会保持原样，但对应的公共模型名会自动转小写存储。你可以：

- 启用需要的模型。
- 修改 public name。
- 删除不需要的模型。

这只是一个初始化辅助，后续仍可在公共模型页面手动调整。

---

## 启用与禁用

- **禁用公共模型**：下游请求该模型名时返回 404。
- **禁用候选**：该候选不再参与路由，但公共模型仍可通过其他候选提供服务。
- **禁用模型组**：下游请求该组名时返回 404。

---

## 最佳实践

- 给同一个真实模型设置多个上游 key 的候选，实现故障转移。
- 给不同供应商的相似模型配置同一个公共模型名，实现供应商透明切换。
- 用模型组封装业务语义（如 `coder`），而不是按供应商分组。
- 定期检查候选健康状态，及时清理失效候选。
